DIFFERENCE IN HTTP/1.1 HTTP/2
YEAR 1997 2015
PROTOCOL TYPE Text based protocol that is in the Human- readable form. It is a binary protocol (HTTP requests are sent in the form of 0s and 1s). Needs to be converted back from binary in order to read it.
KEY FEATURES It supports the resusing of connection, which means that multiple requets and responses could be there in evry TCP (Transfer Control Protocol) connection. Also, pipelining where several resourcescan be reuqested at once from server, but it was hard to impliment due to Head-Of-Line Blocking issues. It uses the method of multiplexing(intiates multiple requests in parallel) over a single TCP connection resources to be delivered are interleaved and arrive at the client almost at the same time. It is done using streams which can be prioritized, can have dependencies and individual flow control. It also provides "Server Push" that send data priorily although client has not yet requested. .
STATUS CODE Introduces a warning header field to carry additional information about the status of a message. Can define 24 status codes, error reporting is quicker and more efficient. Underlying semantics of HTTP such as headers, status codes remains the same.
AUTHENTICATION MECHANISM It uses digest authentication, NTLM authentication. Both are secured mechanism. Security concerns from previous versions will remains same in HTTP/2, but it adds new TLS features like connection error of type Inadequate_Security.
CACHING Expands on the caching support by using additional headers like cache-control, conditional headers like If-Match and by using entity tags. It does not change much in terms of caching. With the server push feature if the client finds the resources are already present in the cache, it can cancel the pushed stream.
PERFORMANCE OPTIMIZATION Spriting, concatenating, inlining, domain sharding are some of the optimizations used Removes the need for unnecessary optimization hacks.
SECURITY SSL is not required but recommended. Digest authentication used in HTTP1.1 is an improvement over HTTP1.0. HTTPS uses SSL/TLS for secure encrypted communication. Though security is still not mandatory, it is mostly encrypted (though it is not enforced) since almost all clients require traffic to be encrypted. It also has some minimum standards, such as minimum key size for encryption. TLS 1.2 etc.
WEB TRAFFIC HTTP/1.1 provides faster delivery of web pages and reduces web traffic as compared to HTTP/1.0. However, TCP starts slowly and with domain sharding (resources can be downloaded simultaneously by using multiple domains), connection reuse and pipelining, there is an increased risk of network congestion. HTTP/2 utilizes multiplexing and server push to effectively reduce the page load time by a greater margin along with being less sensitive to network delays.
HEADER COMPRESSION Headers are sent on every request leading to a lot of duplicate data being sent uncompressed across the wire. Header compression is included by default in HTTP/2 using HPACK.
